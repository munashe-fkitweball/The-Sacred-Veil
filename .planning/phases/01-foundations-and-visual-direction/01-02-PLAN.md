  <action>
For each of the 6 orphan ids, run this audit procedure:

**Step A — Verify the parent is real.** Confirm the parent-section id (e.g. `gath-mere`, `bet-seraphim`, `har-moloch`, `grendels-reach`) exists as a section in `data_sections_*.js` AND has a corresponding entity record in `data.js`. If a parent is missing, surface as a blocker — do NOT proceed with that orphan's classification.

**Step B — Search every canon location for the orphan's display name AND its kebab-case id.** Use grep-equivalent searches:
- `grep -n -i "silveldt" data.js data_sections_*.js tooltips.js page_configs.js`
- `grep -n -i "silveldt\|silveldt" tooltips.js` (case-insensitive on display name; exact on id)
- Repeat for: `the[ -]drift`, `the[ -]rot`, `third[ -]furnace`, `tide[ -]reach[ -]wharf`, `anvilhold`

For each hit, record:
- File and line number
- The exact verbatim text of the surrounding context (the full sentence or block)
- A judgment: is this **material** description (≥2-3 sentences of substantive content; a coherent paragraph that could stand as a section's `blocks: [{type: 'p', text: '…'}]`) OR is this a **namedrop** (a single mention in passing, e.g. "with Anvilhold to the south, Silveldt across the bay")?

**Step C — Classify.**
- If at least ONE hit is material AND can be lifted as-is into a section's `blocks` array WITHOUT requiring any new transitions or smoothing sentences → **Mechanism A**.
- Else → **Mechanism B**.

**Step D — Record uncertainty.** If a hit is borderline (looks substantive but might need a transition sentence to read as a standalone section), classify it as **Mechanism B** and add a note in the Uncertainty column. Do NOT classify as Mechanism A unless the lift is clean. Bias toward B because B is zero-prose-risk.
